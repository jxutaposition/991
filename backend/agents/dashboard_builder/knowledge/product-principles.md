# Principles — Product Design

## Define the row unit before building

Before any enrichment, lookup, or scoring column: what does one row represent? If the unit doesn't match the data (e.g., enrichment is per-post but the row is per-expert), restructure first. Wrong row unit compounds into broken outputs.

## Measurement must be operational, not aspirational

Only propose tracking what you can actually instrument today. Show status clearly: what's live, what's a gap, what's blocked on access. Aspirational metrics don't drive behavior — operational ones do.

## Revenue is the primary north star, but never alone

Referral ARR is the anchor metric. But a program that only optimizes revenue will hollow out. Layer in content signals (impressions, posts), community signals (CSAT, participation), and support load as health checks.

## Gaps in data should be visible, not hidden

If a data source isn't flowing yet, mark it as a gap explicitly. Track what's missing with the same rigor as what's present. Hidden gaps become silent failures.

## Dashboard design follows program logic

Don't put every metric on one screen. Internal view (full stats) and external view (public points-only) serve different audiences. MRR data stays internal — not because it's secret, but because visibility creates support friction for edge cases (e.g., complimentary plans).

## Point values need calibration against real data before launch

Set provisional values, pull real distribution data, then calibrate. Don't commit to a scoring system before you know what ranges the data actually produces.
