# Gap Workstream Prompts

These are self-contained session prompts for closing each gap between the 99percent product thesis and the current codebase. Each prompt is designed to be dropped into a fresh Cursor/Claude conversation working inside this repo.

The prompts are ordered by priority. Each one should produce a **system design document** (like SD-001 through SD-003) plus an **implementation plan** that respects the existing architecture.

**Ground rules for every workstream:**
- The architecture must be a living, breathing system -- not a one-shot build. Design for continuous evolution.
- Postgres is the source of truth. Expert and client knowledge is ingested via workspace uploads and the knowledge corpus (`knowledge_documents` / RAG), not committed client-specific files in this repo.
- Backend is Rust/Axum (see `backend/src/`). Frontend is Next.js 15 / React 18 / Tailwind (see `frontend/src/`).
- Anthropic Claude is the LLM (called via `backend/src/anthropic.rs`). No other LLM providers.
- Existing system designs: `system_design/SD-001_agent_config_management.md`, `SD-002_integrations_and_credentials.md`, `SD-003_orchestrator_primitives_and_learning.md`.
- The overlay system (base/expert/client/project scoping) is a core primitive. New features should use it, not bypass it.

## WORKSTREAM 1: Plan as Rich, Editable Document

### Context

The plan is the fundamental primitive of this entire product. The thesis states: "The basic unit of information is a plan." Currently, a plan is a JSON array of `PlannedNode { agent_slug, task_description, depends_on }` generated by `backend/src/planner.rs` and displayed as colored boxes on a DAG canvas in `frontend/src/components/execution-canvas.tsx`. Users can edit per-node task descriptions and swap agents in `frontend/src/components/inspector-panel.tsx` before approving.

The thesis envisions something radically richer: a plan that feels like a system design document crossed with a Google Doc, where each node contains mockups, technical specs, I/O schemas, links to prior artifacts, optionality annotations, and blocker callouts. The user edits it conversationally -- highlight text, type a comment, the agent responds and updates the plan inline.

### Task

Design and implement a system where the plan is a **rich, structured document** that serves as both the specification and the execution blueprint. This is the "cursor for GTM" moment -- the plan must feel like a living design document, not a task list.

### Key files to read first

- `backend/src/planner.rs` -- current plan generation (PlannedNode, plan_execution, plan_to_execution_nodes)
- `backend/src/routes.rs` -- execution_create handler, node PATCH endpoint
- `frontend/src/app/execute/[session_id]/page.tsx` -- session page with approval flow
- `frontend/src/components/execution-canvas.tsx` -- DAG visualization (buildDag, NodeBox)
- `frontend/src/components/inspector-panel.tsx` -- node detail/edit panel
- `backend/agents/master_orchestrator/prompt.md` -- how the orchestrator decomposes work
- `system_design/SD-003_orchestrator_primitives_and_learning.md` -- orchestrator architecture

### What "rich plan" means concretely

Each plan node should evolve from a single `task_description` string to a structured object containing:

- **Architecture section**: what this component does, how it connects to others, data flow in/out
- **Technical spec**: specific tools/APIs involved, configuration details, expected schemas
- **Visual mockup reference**: for UI components, a description or link to what it should look like
- **I/O contract**: explicit input schema (what upstream provides) and output schema (what downstream expects), with examples
- **Prior artifact references**: links to past deliverables from the knowledge corpus that are relevant (e.g., "see the tiering system built for Client X in project Y")
- **Optionality**: where multiple implementation paths exist, enumerate them with tradeoffs (e.g., "Clay table vs. direct API -- Clay is faster to set up, direct API gives more control")
- **Blockers**: missing API keys, manual steps required, decisions needing expert review, tables that must be created first
- **Acceptance criteria**: how to validate this node's output

### Conversational editing

The user must be able to:
1. View the full plan as a scrollable document (not just a DAG)
2. Select/highlight any text within a node's content
3. Type a comment or question about the highlighted section
4. The agent responds in a threaded conversation attached to that section
5. The agent can update the plan content based on the conversation
6. Changes are versioned -- the user can see what changed and why

Think about how Cursor's inline chat works, or Google Docs commenting. The plan is the document. Comments are the conversation. The agent is a collaborator.

### Blocker identification

During plan generation, the agent should proactively identify and flag:
- Missing credentials (cross-reference `integrations` and `client_credentials` tables)
- Manual prerequisites (tables that need to be created, accounts that need to be set up)
- Decision points that require expert judgment (where the agent isn't confident enough to choose)
- External dependencies (waiting on client data, third-party access)

These should be visually distinct in the plan UI -- not buried in prose.

### Constraints

- The plan must still be executable by the work queue. The rich document is the human-facing representation; the execution graph (nodes + dependencies) is derived from it.
- Plan generation should use the master orchestrator's thinking (not the one-shot planner) to produce deeper analysis.
- The plan document must be storable in Postgres (JSONB on `execution_sessions.plan` or a new table).
- The conversational editing must support real-time collaboration (SSE or WebSocket) -- the agent and user are co-editing.

### Deliverables

1. A system design document (SD-004) defining the plan document schema, the editing protocol, and the UI architecture
2. Database migration for the rich plan schema
3. Backend: plan generation that produces rich nodes, comment/conversation endpoints, plan versioning
4. Frontend: plan document view (alongside or replacing the DAG canvas), inline commenting UI, blocker visualization
5. A clear spec for how the rich plan maps to execution nodes when approved

## WORKSTREAM 2: Knowledge Corpus Ingestion and RAG Pipeline

> **Superseded by [SD-006: Knowledge and Data Architecture](SD-006_knowledge_and_data_architecture.md).** The design below has been implemented and extended with hybrid search (vector + BM25), contextual retrieval (Anthropic technique), Claude reranking, and neighbor chunk expansion. See SD-006 for the canonical architecture. The original prompt is preserved below for historical reference.

### Context

The thesis states: "These best GTM engineers are now building extensive knowledge banks in markdowns of meetings, designs, contracts, decision points, claude conversation summaries. We can push this further by collecting browser data." And: "Expert/team uploads full knowledge bank always... direct access to the file system, or a connector to always consume the new info."

Expert knowledge is expected to live primarily in the **uploaded knowledge corpus** (and connected sources), not in git-tracked client folders. The browser extension (`extension/`) captures workflow events (clicks, navigation, screenshots) and streams them to `POST /api/observe/session/:id/events`. The overlay system stores learned lessons in Postgres. Historically some teams kept a local markdown corpus; that content should be **uploaded** for RAG rather than baked into agent prompts. Gaps that may remain:

- Bulk upload mechanism for a knowledge bank
- Incremental sync from an external source
- Conversation transcript ingestion (from Claude Code, Cursor, or any chat)
- Semantic indexing over the full corpus
- RAG retrieval during agent execution or junior troubleshooting

### Task

Design and implement a **knowledge ingestion pipeline** that continuously captures, indexes, and serves expert knowledge. This is what makes the expert "always available" to juniors and agents. It must be a living system that grows richer over time without manual curation.

### Key files to read first

- `frontend/src/app/knowledge/page.tsx` -- knowledge upload and browse UI
- `backend/src/routes.rs` -- `knowledge_document_upload`, search, and related APIs
- `backend/src/agent_runner.rs` -- how agents get context injected (upstream outputs, client context, credentials, skill overlays)
- `backend/migrations/001_agent_catalog.sql` -- existing pgvector embeddings for agent catalog
- `backend/src/feedback.rs` -- synthesize_feedback uses Claude to analyze signals
- `extension/src/background.ts` -- browser extension streaming events + screenshots to backend
- `system_design/SD-003_orchestrator_primitives_and_learning.md` -- overlay system and learning

### What the ingestion pipeline must handle

**Source types:**
1. **Bulk upload**: expert uploads a directory of markdown/docs/PDFs. System parses, chunks, embeds, indexes.
2. **Incremental sync**: expert points at a local directory or cloud storage (Google Drive, Notion export). System detects new/changed files on a schedule or webhook.
3. **Conversation transcripts**: Claude Code conversation logs, Cursor agent transcripts, meeting transcripts. Parse into structured knowledge (decisions made, artifacts referenced, lessons learned).
4. **Browser extension data**: already partially built. Enhance to extract higher-level patterns from click/nav sequences (e.g., "expert spent 20 minutes debugging Clay table X, then changed column Y").
5. **Execution artifacts**: outputs from completed agent sessions. These are already in Postgres but not indexed for retrieval.

**Processing pipeline:**
1. **Parse**: extract text from markdown, PDF, docs, conversation logs
2. **Chunk**: split into semantically meaningful chunks (not just by token count -- respect document structure)
3. **Classify**: tag each chunk with metadata (source type, client, project, topic, date)
4. **Embed**: generate embeddings (pgvector is already in the stack)
5. **Index**: store in a queryable format that supports both semantic search and filtered retrieval
6. **Link**: connect chunks to entities (clients, projects, skills, artifacts) in the existing data model

**Retrieval interface:**
1. **Agent-facing**: during execution, agents can call a `search_knowledge` tool that does RAG over the corpus, scoped to the relevant client/project/expert
2. **Junior-facing**: when a junior is stuck, they can ask a question and get answers grounded in the expert's knowledge
3. **Plan-generation-facing**: when generating a plan, the planner should automatically pull relevant prior artifacts and reference them in plan nodes (see Workstream 1)

### Drift detection

The system must detect when the expert's local corpus has new information that hasn't been ingested:
- If using sync: compare file hashes/timestamps
- If using upload: prompt the expert periodically ("You have 12 new files since last upload")
- Surface staleness in the UI: "Knowledge last synced 3 days ago"

### Data model considerations

- Where does ingested knowledge live? Probably a new `knowledge_chunks` table with embeddings, metadata, source tracking
- How does it relate to overlays? Ingested knowledge is raw material; overlays are curated lessons. The RAG pipeline serves raw knowledge; the overlay system serves distilled wisdom. They complement, not replace, each other.
- Scoping: knowledge chunks should be scoped (expert, client, project) just like overlays
- Deduplication: the same insight may appear in multiple sources (meeting note + conversation + execution output). Detect and link duplicates.

### Constraints

- pgvector is already a dependency. Use it for embeddings.
- The ingestion pipeline should be async (background jobs, not blocking the API).
- Support flexible folder and metadata from uploads; do not require a single fixed on-disk layout in the repo.
- Start with markdown as the primary format. PDF/doc parsing can come later.
- The RAG retrieval must be fast enough to be called mid-agent-execution without noticeable delay.

### Deliverables

1. System design document (SD-005) defining the ingestion pipeline, data model, retrieval interface, and sync protocol
2. Database migration for `knowledge_chunks` (or similar) with pgvector embeddings
3. Backend: ingestion endpoints (upload, sync trigger), background processing pipeline, `search_knowledge` tool for agents
4. Frontend: knowledge management UI (upload, sync status, browse indexed content, staleness indicators)
5. Integration with agent_runner.rs: agents can retrieve knowledge during execution

## WORKSTREAM 3: Expert/Junior Role Separation and Plan Handoff

### Context

The thesis states: "We will solve labor shortages by enabling small agencies to relentlessly hire junior engineers who are able to replicate the work of top 1% experts." And: "The expert will review the plan, give edits/feedback... When it is approved, it can be given back to the junior. The junior now has a living, breathing spec."

Currently, the system has an `experts` table and `engagements` but operates as a single-user experience. The person who creates a session is the person who approves and monitors it. There is no concept of a "junior" who receives an approved plan and executes it alongside the agent, with access to the expert's knowledge corpus.

### Task

Design and implement a **role-based workflow** where experts create and review plans, juniors execute them alongside agents, and the expert's knowledge is always accessible to the junior through the agent. This is the core scaling mechanism of the product.

### Key files to read first

- `backend/src/routes.rs` -- auth handlers, execution CRUD, all API endpoints
- `frontend/src/lib/auth-context.tsx` -- current auth model (Google sign-in, JWT, apiFetch)
- `frontend/src/components/nav.tsx` -- workspace/client switching
- `frontend/src/app/execute/[session_id]/page.tsx` -- session flow (create -> approve -> execute -> monitor)
- `backend/migrations/008_learning.sql` -- clients, workflows, agent_run_history tables
- `backend/migrations/` -- look at the users/auth migrations for current user model
- `system_design/SD-003_orchestrator_primitives_and_learning.md` -- how context flows through the system

### Role model

**Expert:**
- Creates execution sessions from contract/deliverable input
- Reviews and edits the generated plan (Workstream 1)
- Approves the plan
- Optionally delegates to a junior
- Reviews completed work
- Their knowledge corpus powers the agent (Workstream 2)
- Can intervene at any point during junior execution

**Junior:**
- Receives an approved plan as a "work order"
- Executes it step-by-step alongside the agent
- Can ask the agent questions that are answered from the expert's knowledge corpus
- Can flag blockers that get escalated to the expert
- When stuck, the agent provides guidance grounded in expert artifacts and past decisions
- Their interactions become new feedback signals that improve the system

**Client (future):**
- Read-only view of plan and progress (Workstream 7 -- not in scope here but design for it)

### Plan handoff flow

1. Expert creates session, plan is generated
2. Expert reviews, edits, approves the plan
3. Expert assigns the plan to a junior (new: assignment endpoint, junior selection UI)
4. Junior sees the plan in their dashboard with full context
5. Junior starts execution -- the agent runs, the junior monitors and intervenes
6. When the agent is blocked or the junior is stuck, the junior can:
   - Ask the agent a question (existing reply mechanism)
   - The agent answers using RAG over the expert's knowledge corpus (requires Workstream 2)
   - Escalate to the expert (new: escalation mechanism, expert notification)
7. Expert can see all active juniors' work and intervene at any time
8. Junior completes execution, expert reviews the output

### What the junior interface needs

- **Guided execution view**: the plan as a checklist/walkthrough, not just a DAG. Each step has instructions, expected outcomes, and "what to do if stuck" guidance.
- **Knowledge-augmented chat**: when the junior talks to the agent, the agent has retrieval access to the expert's full history. This is how the junior "has unfettered access to direct corpuses of knowledge from an expert."
- **Escalation button**: flag a step as blocked with a note to the expert. Expert gets notified. Expert can respond inline or take over.
- **Learning capture**: every junior interaction (questions asked, corrections made, blockers hit) becomes a feedback signal. The system learns what juniors struggle with.

### Data model changes

- User roles: expert, junior, admin (extend existing users table or add a roles table)
- Team/org model: an expert has juniors. A junior belongs to one or more experts.
- Assignment: an execution session can be assigned to a junior after approval
- Escalation: a mechanism for juniors to escalate to experts (could be a simple message thread on the session, or a dedicated escalation table)

### Constraints

- The existing auth system (Google sign-in + JWT) should be extended, not replaced.
- The expert must be able to see everything the junior sees, plus more (full feedback signals, raw agent conversations, etc.).
- The junior's interactions must feed into the feedback pipeline (existing `feedback_signals` table).
- Design for the case where one expert has 5-10 juniors working on different client projects simultaneously.

### Deliverables

1. System design document (SD-006) defining the role model, handoff flow, junior interface, and escalation mechanism
2. Database migrations for roles, assignments, escalation
3. Backend: role-based auth middleware, assignment endpoints, escalation endpoints, knowledge-augmented reply (depends on Workstream 2)
4. Frontend: expert dashboard (see all juniors' work), junior dashboard (assigned plans), junior execution view with guided walkthrough, escalation UI
5. Integration with the feedback pipeline: junior interactions as learning signals

## WORKSTREAM 4: Continuous Monitoring, Alerting, and Repair

### Context

The thesis states: "The agent is able to continuously monitor, alert, and repair this technical stack for the downstream client -- the agency is able to maintain a continuous contract with low effort, and the client receives continued support and up to date expertise."

Currently, agents execute once and complete. The `workflows` table with `workflow_steps` exists for recurring patterns but there is no persistent monitoring capability. Once a plan is executed and completed, the system has no ongoing relationship with the built artifacts.

### Task

Design and implement a **monitoring and maintenance system** where completed plans become monitored systems. Agents periodically check that built artifacts are still functioning, alert on anomalies, and can automatically repair issues using the original plan as reference.

### Key files to read first

- `backend/src/work_queue.rs` -- how agent execution is scheduled and run (polling, concurrency, FOR UPDATE SKIP LOCKED)
- `backend/src/workflow.rs` -- existing workflow/scheduling infrastructure
- `backend/src/agent_runner.rs` -- agent execution loop, tool calls, result persistence
- `backend/src/tools.rs` -- available tools (http_request, browser, etc.)
- `backend/migrations/008_learning.sql` -- workflows, workflow_steps tables
- `system_design/SD-003_orchestrator_primitives_and_learning.md` -- validation layers

### What monitoring means for GTM systems

GTM automations are pipelines of interconnected tools. Common failure modes:
- **Data pipeline breaks**: a Clay table stops enriching because an API key expired, a column was renamed, or a rate limit changed
- **Integration drift**: a webhook URL changes, an OAuth token expires, a third-party API version is deprecated
- **Data quality degradation**: enrichment accuracy drops, dedup logic lets duplicates through, scoring produces anomalous distributions
- **Dashboard staleness**: Supabase data stops flowing, Lovable dashboard shows stale data
- **Workflow failures**: n8n workflows error out silently, scheduled jobs stop running

### Monitor definitions

A monitor is a persistent check attached to a completed execution plan. Each monitor specifies:
- **What to check**: an HTTP endpoint to probe, a data query to run, a dashboard to screenshot and compare
- **How often**: cron schedule (every hour, daily, weekly)
- **Acceptable state**: expected response code, expected data shape, acceptable value ranges, visual similarity threshold
- **Alert threshold**: how many consecutive failures before alerting
- **Auto-repair policy**: whether the agent should attempt to fix the issue automatically, or just alert

Monitors should be **generated from the plan**. When a plan node describes building a Clay->Supabase pipeline, the system should automatically propose monitors for that pipeline (e.g., "check that new rows appear in Supabase daily", "verify Clay enrichment accuracy weekly").

### Alert pipeline

1. Monitor check fails -> create an `alert` record
2. If auto-repair is enabled and confidence is high -> spawn a repair agent with the original plan node as context
3. If auto-repair succeeds -> resolve alert, log the fix as a feedback signal
4. If auto-repair fails or isn't enabled -> notify the assigned junior and/or expert
5. Alert dashboard: per-client health view showing all active monitors, recent alerts, system health score

### Repair agents

A repair agent is a specialized agent that:
- Has the original plan node as context (what was built, how, why)
- Has access to the knowledge corpus (how similar issues were resolved before)
- Has the same tools/credentials as the original builder agent
- Attempts to diagnose and fix the issue
- Reports what it found and what it did
- If it can't fix it, produces a detailed diagnostic for the junior/expert

### Constraints

- Monitoring runs on the server (cron-like, not dependent on any user being online)
- Must work with the existing credential system (monitors use stored client credentials)
- Monitor execution should use the same work queue infrastructure (or a parallel one)
- Storage: alerts, monitor definitions, check history should all be in Postgres
- Must be pausable per-client (if a client churns, stop their monitors)

### Deliverables

1. System design document (SD-007) defining monitor definitions, scheduling, alert pipeline, repair agents
2. Database migrations for monitors, alerts, check_history
3. Backend: monitor CRUD, scheduler (extend workflow infrastructure or build parallel), alert pipeline, repair agent spawning
4. Frontend: monitor management UI, alert dashboard, per-client health view
5. Plan integration: automatic monitor proposal during plan generation (suggested monitors extracted from plan nodes)

## WORKSTREAM 5: Cross-Project Knowledge Graph and Artifact Registry

### Context

The thesis states: "Future agents have access to all previous agents/project artifacts -- this is the interconnected knowledge work that exposes their GTM tech stack as operable/navigatable 'code'."

Currently, execution outputs are stored as JSONB on `execution_nodes.output`. Overlays promote patterns across projects. Past work may also be captured in the **uploaded knowledge corpus** (briefs, methodology, deliverable notes). But there is no explicit registry of artifacts, no cross-project linking, and no way for an agent to "navigate" the full history of what's been built.

### Task

Design and implement an **artifact registry and knowledge graph** that makes every deliverable, decision, and component referenceable and navigable -- like a codebase, but for GTM systems.

### Key files to read first

- `backend/migrations/003_execution.sql` -- execution_sessions, execution_nodes, execution_events
- `backend/src/agent_runner.rs` -- how outputs/artifacts are persisted
- Uploaded knowledge documents -- client briefs, program docs, access runbooks (scoped per tenant/project)
- `backend/migrations/008_learning.sql` -- clients, workflows, agent_run_history
- `system_design/SD-003_orchestrator_primitives_and_learning.md` -- overlay system

### What artifacts are in this context

An artifact is any concrete deliverable or component produced during plan execution:
- A Clay table configuration (columns, enrichments, filters)
- An n8n workflow (nodes, connections, triggers)
- A Supabase schema (tables, views, RLS policies)
- A Lovable dashboard (routes, components, data sources)
- A Notion workspace structure (databases, pages, templates)
- A scoring/tiering system design (tiers, criteria, thresholds)
- A data pipeline definition (source -> transform -> destination)
- A webhook configuration (URL, payload schema, auth)

### The knowledge graph

Artifacts exist in relationships:
- **Depends on**: this dashboard depends on this Supabase table, which depends on this Clay enrichment pipeline
- **Supersedes**: this v2 tiering system replaces the v1
- **References**: this plan node referenced this prior artifact as a template
- **Shared by**: this Clay table is used by both Client A's pipeline and Client B's pipeline
- **Owned by**: this artifact was built for Client X, Project Y, by Expert Z
- **Monitored by**: this artifact has these active monitors (Workstream 4)

### How agents use the graph

1. **During plan generation**: "We're building a tiering system for a new client. Here are 3 prior tiering systems we've built, with their designs and what worked/didn't."
2. **During execution**: "This Clay table pattern is similar to one we built for Client X. Here's how we handled the edge case with VP titles."
3. **During troubleshooting**: "This n8n workflow broke. Here's how a similar workflow was repaired last month."
4. **During maintenance**: "Client X's system uses components shared with Client Y. A change to this shared component affects both."

### Data model

- `artifacts` table: id, type (clay_table, n8n_workflow, supabase_schema, etc.), name, description, metadata JSONB, source_session_id, source_node_id, client_id, project_id, expert_id, created_at, updated_at
- `artifact_relationships` table: from_artifact_id, to_artifact_id, relationship_type (depends_on, supersedes, references, etc.)
- `artifact_versions` table: artifact_id, version, snapshot JSONB, change_summary
- Embeddings on artifacts for semantic search

### Constraints

- Artifact extraction should be automatic where possible (parse execution outputs for recognizable structures)
- Manual artifact registration should also be supported (expert says "this Clay table is an artifact worth tracking")
- The graph should be queryable by agents via a tool (e.g., `search_artifacts`, `get_related_artifacts`)
- Must integrate with the knowledge corpus (Workstream 2) -- artifacts are a type of knowledge
- Must integrate with monitoring (Workstream 4) -- monitored artifacts link to their monitors

### Deliverables

1. System design document (SD-008) defining the artifact registry, knowledge graph schema, extraction pipeline, and query interface
2. Database migrations for artifacts, artifact_relationships, artifact_versions
3. Backend: artifact CRUD, automatic extraction from execution outputs, graph query endpoints, agent tools for artifact search
4. Frontend: artifact browser (graph visualization or list view), artifact detail page, cross-project navigation
5. Integration points: plan generation references prior artifacts, agents query the graph during execution

## WORKSTREAM 6: Contract/Deliverable Structured Input

### Context

The thesis says: "User copy-pastes the contract details (requirements, context, deliverables)." Currently, `POST /api/execute` takes a freeform `request_text` string with optional `client_id` and `model`.

### Task

Design a **structured input flow** for creating execution sessions from contract/deliverable specifications. This should feel like onboarding a new project -- not just typing a prompt.

### Key files to read first

- `backend/src/routes.rs` -- execution_create handler
- `backend/src/planner.rs` -- how request_text feeds into plan generation
- `frontend/src/app/execute/page.tsx` -- session creation UI
- Example client briefs or scoping templates **as uploaded knowledge** for a workspace (not hardcoded in-repo examples)

### What structured input means

Instead of a single text box, the input flow should support:

1. **Paste raw contract/brief**: the user pastes unstructured text, and the system parses it into structured sections using Claude
2. **Or fill structured form**: requirements, context, deliverables, timeline, access/credentials needed, constraints
3. **Client context auto-loading**: when a client is selected, automatically pull in their brief, existing artifacts, known credentials, prior projects
4. **Deliverable mapping**: each deliverable becomes a trackable item that maps to plan nodes
5. **Gap identification**: after parsing, surface what's missing (unclear requirements, missing access, ambiguous scope)

### Constraints

- Must support both the "paste and parse" and "structured form" paths
- The parsed/structured input should be stored (new fields on execution_sessions or a related table)
- Plan generation should use the structured input, not just the raw text
- The structured input should be editable after creation (before plan generation)

### Deliverables

1. System design document (SD-009) defining the input schema, parsing logic, and UI flow
2. Backend: structured input fields, Claude-powered parsing endpoint, enhanced plan generation that uses structured input
3. Frontend: new session creation flow with paste+parse and structured form options, gap identification UI

## WORKSTREAM 7: Customer-Facing Plan Visibility

### Context

The thesis states: "The plan will continue being the basic unit of information passed between and agreed upon between people (even GTM and their customers. All can have live access to what is being built)."

### Task

Design a **customer-facing read-only view** of plans and execution progress. Customers (the agency's clients) should be able to see what's being built for them without needing full platform access.

### Key files to read first

### What customer visibility means

1. **Shareable link**: expert generates a share link for a plan/session (like Google Docs sharing)
2. **Read-only view**: customer sees the plan document, execution progress, deliverable status
3. **No sensitive data**: filter out internal notes, agent conversations, credential details, internal feedback

### Constraints

- No authentication required for shared links (or simple token-based auth)
- The shared view must be a separate route/component that doesn't expose internal platform UI
- Must be designable independently of the full platform redesign
- Rate limiting and expiry on shared links

### Deliverables

1. System design document (SD-010) defining the sharing model, customer view, and access control
2. Backend: share link generation, token-based access, filtered data endpoints
3. Frontend: customer-facing plan/progress view (clean, branded, read-only)

## WORKSTREAM 8: Architecture Decision Document

### Context

The thesis raises the fundamental question: build directly on Claude Code/Cursor, or build your own system that hits the Claude API? The codebase analysis shows the system is already fully self-hosted. This workstream documents the decision and its implications.

### Task

Write a formal architecture decision record (ADR) that:

1. **States the decision**: own stack with Claude API, Claude Code/Cursor as optional data connectors
2. **Documents the reasoning**: multi-tenancy requirements, persistent execution, learning loops, monitoring -- all require server-side state that Claude Code can't provide
3. **Defines the integration points**: where Claude Code/Cursor CAN plug in (local knowledge sync, conversation transcript ingestion, optional local agent for file system access)
4. **Maps the data flow**:
   - Expert's local environment (markdown files, Claude Code conversations, browser activity) -> ingestion connectors -> Lele server
   - Lele server (plan generation, execution, learning, monitoring) -> Lele frontend
   - Lele frontend -> expert view, junior view, customer view
5. **Identifies what we give up**: Claude Code's native thinking UX, local file system access, Cursor's inline editing. And how we compensate (our own plan editing UX, knowledge upload, browser extension).
6. **Future-proofs**: if Claude Code adds an API/SDK for external orchestration, how would we integrate?

### Key files to read first

- `system_design/SD-001_agent_config_management.md`
- `system_design/SD-002_integrations_and_credentials.md`
- `system_design/SD-003_orchestrator_primitives_and_learning.md`

## Sequencing and Dependencies

```
WORKSTREAM 8 (Architecture Decision)
    ↓ (should be done first -- frames everything)
WORKSTREAM 2 (Knowledge RAG)
    ↓ (required by 1 and 3)
WORKSTREAM 1 (Rich Plan) ←→ WORKSTREAM 6 (Structured Input)
    ↓                              ↓
WORKSTREAM 3 (Expert/Junior Roles)
    ↓
WORKSTREAM 4 (Monitoring) ←→ WORKSTREAM 5 (Knowledge Graph)
    ↓
WORKSTREAM 7 (Customer Visibility)
```

- **Workstream 8** first: lock the architecture decision so everything else is on solid ground
- **Workstream 2** next: the RAG pipeline is a dependency for both the rich plan (artifact references) and expert/junior roles (knowledge-augmented chat)
- **Workstreams 1 and 6** can be done in parallel (rich plan + structured input feed into each other)
- **Workstream 3** depends on 1 and 2 (needs the rich plan for handoff, needs RAG for junior knowledge access)
- **Workstreams 4 and 5** can be done in parallel after 3 (monitoring and artifact graph are related but independent)
- **Workstream 7** last (customer visibility is a layer on top of everything else)
