# SD-005: Plan Document and Conversation Architecture

## Decision

The orchestrator is the user's primary conversational interface -- their "Claude Code." The plan is a living system design document, not a task list. Conversations are hierarchical, mirroring the execution hierarchy. The user is always in control and can interject at any time.

---

## Part 1: Philosophy and Mental Model

### The Orchestrator Is the User's Claude Code

The master orchestrator is not a background service that silently coordinates agents. It is a visible, addressable collaborator -- the user's local AI partner for the entire session. The user watches it think, steers it with messages, and can interject at any time.

This means:
- The orchestrator's thinking stream is always visible in the bottom panel (the primary conversation interface).
- The user's input box is always enabled. Messages are never blocked.
- When the orchestrator is busy (mid-LLM-call, mid-planning, mid-coordination), user messages are queued and processed at the next natural break point.
- The orchestrator acknowledges queued messages and incorporates them into its next action.

The mental model is: you are pair-working with the orchestrator. You describe what you want. It thinks out loud, decomposes the work, delegates to specialized agents, and reports back. You can interrupt, redirect, or ask questions at any point.

### The Plan Is a Living Document

A plan starts as a system design document generated by the orchestrator's thinking. It serves as the execution blueprint during the approval phase. As execution produces results, the plan evolves -- nodes gain outputs, artifacts, and status updates. After execution, the plan persists as living system documentation.

Each plan node is a structured specification, not a description string. A node contains:
- **Architecture**: what this component does, how it connects to others, data flow
- **Technical spec**: approach, tools/platforms involved, configuration
- **I/O contract**: explicit input and output schemas with sources
- **Optionality**: alternative implementation paths with tradeoffs
- **Blockers**: missing credentials, manual prerequisites, decision points
- **Acceptance criteria**: how to validate the node's output
- **Mockup reference**: for UI components, a description of expected visual output
- **Prior artifacts**: references to relevant past deliverables (populated by RAG)

This structure makes the plan reviewable, editable, and useful as a reference long after execution completes.

### Conversations Are Hierarchical

The conversation model mirrors the execution hierarchy:

```
Orchestrator conversation (top-level)
├── Node A agent conversation
│   ├── Section thread: architecture
│   └── Section thread: I/O contract
├── Node B agent conversation
│   └── Section thread: technical spec
└── Node C agent conversation
```

- **Orchestrator conversation**: The main stream. Planning thoughts, coordination decisions, status updates, user messages.
- **Node conversations**: Each node's agent has its own conversation -- reasoning, tool calls, outputs, user replies when the agent asks.
- **Section threads**: Anchored to specific sections of a node's rich description (e.g., "architecture", "io_contract"). For focused discussion about a particular aspect of the spec.

### The User Is Always in Control

- Messages to the orchestrator are never blocked. They queue when the orchestrator is busy and are processed at the next break point.
- The user can steer planning ("use Clay instead of direct API"), redirect execution ("skip that node"), flag issues ("the API key is wrong"), or ask questions ("what's the data flow?") without waiting for a "your turn" signal.
- Queued messages appear in the conversation stream with a "queued" indicator. The orchestrator sees them in order and responds before continuing its planned work.

## Part 2: Streaming Plan Generation Flow

### Flow (Implemented)

```
User types prompt → clicks "Build Plan"
→ POST /api/execute inserts session with status "planning", returns { session_id } IMMEDIATELY
→ router.push(/execute/:session_id) → session status is "planning"
→ SSE connects → planner_progress events stream in real-time
→ Plan nodes appear on canvas as they're generated
→ Status transitions to "awaiting_approval" → user reviews and approves
```

> This streaming plan generation flow is fully implemented. The SSE channel is created in `execution_create`, not `execution_approve`.

### Implementation

1. **Async plan generation**: `execution_create` inserts the session row with status `planning` and returns `session_id` immediately. A `tokio::spawn` background task runs the planner.
2. **SSE from the start**: The EventBus channel is created in `execution_create`, not in `execution_approve`. The frontend connects SSE as soon as it lands on the session page.
3. **Planning events**: The background task emits events through the EventBus:
   - `planner_progress`: status updates at key checkpoints (analyzing request, loading catalog, calling LLM, parsing response)
   - `planner_thinking`: extended thinking deltas from the LLM (when streaming is supported)
   - `plan_ready`: final event when the plan is complete, with the full plan payload
4. **Status transition**: When planning completes, the background task updates the session status to `awaiting_approval` and inserts preview nodes. The frontend detects this via the `plan_ready` event or session refetch and transitions the UI to the approval view.

## Part 3: Chat Architecture

### Bottom Panel: The Unified Conversation Interface

The bottom panel is the primary conversation interface for the entire session.

**Default tab: "Orchestrator"**
- Shows the master orchestrator's thinking stream, messages, and coordination decisions.
- During `planning`: shows planner thinking/progress in real-time.
- During `awaiting_approval`: shows the completed plan summary; user can ask questions or request changes.
- During `executing`: shows orchestrator coordination (delegating to agents, receiving results, making decisions).
- Input box is always enabled.

**Per-node tab: "Node: {name}"**
- Appears when a node is clicked on the canvas or in the document view.
- Shows that node's agent conversation: reasoning, tool calls, outputs.
- User can reply if the node is in `awaiting_reply` status.
- Only one node tab at a time. Clicking a different node switches to it. Clicking away returns to Orchestrator.

### Queue Semantics

> **Status: Not implemented.** Messages can be sent to the orchestrator via `POST /api/execute/:session_id/nodes/:node_id/reply`, but there is no queue mechanism, no break-point processing, and no `user_message_queued` SSE event. User messages are only processed when the agent is in `awaiting_reply` status.

**Planned behavior:** When the user sends a message while the orchestrator is busy:

1. Message is persisted to `node_messages` on the master node with role `user`.
2. A `user_message_queued` event is emitted via SSE.
3. Frontend shows the message in the conversation stream with a subtle "queued" indicator.
4. The orchestrator checks for unprocessed user messages at each natural break point.
5. When the orchestrator processes a queued message, it acknowledges it and may adjust its plan/actions accordingly.

### Side Channel ("Quick Ask")

A lightweight floating chat for questions that shouldn't enter the orchestrator's context:
- Uses the `ProjectChat` component (`POST /api/projects/:id/chat` or `/api/execute/:id/chat`).
- Stateless Q&A. Messages are not injected into the orchestrator's context or execution flow.
- Examples: "What's the Clay API rate limit?", "Remind me what the client's Supabase URL is."

### Per-Node Chat Accessibility

Each node's conversation is accessible in **all** view modes:
- **Canvas mode**: via InspectorPanel (right side) and bottom panel node tab.
- **Document mode**: via bottom panel node tab (click a node section to switch).
- **Split mode**: via bottom panel node tab.

## Part 4: Plan Document Schema

### Project-Level Description

Table: `project_descriptions`

| Column | Type | Description |
|--------|------|-------------|
| id | UUID | Primary key |
| project_id | UUID | FK to projects |
| title | TEXT | Short project title |
| summary | TEXT | 1-3 paragraph system overview |
| architecture | JSONB | System architecture narrative (overview, patterns) |
| data_flows | JSONB | Array of { from, to, description } |
| integration_map | JSONB | Which platforms, how they connect |
| version | INT | Auto-incrementing version number |

### Node-Level Description

Column: `execution_nodes.description` (JSONB)

```json
{
  "display_name": "Expert Scoring Pipeline",
  "architecture": {
    "purpose": "Scores and tiers partners based on engagement and revenue metrics",
    "connections": ["clay", "supabase"],
    "data_flow": "Clay enrichment → composite scoring → Supabase upsert"
  },
  "technical_spec": {
    "approach": "Pull enrichment data, compute weighted composite score, write tiers",
    "tools": ["clay", "supabase"],
    "configuration": {}
  },
  "io_contract": {
    "inputs": [{ "name": "partner_list", "source": "clay_enrichment", "schema": {} }],
    "outputs": [{ "name": "scored_partners", "schema": {} }]
  },
  "optionality": [
    { "decision": "Scoring method", "tradeoffs": "Weighted average vs ML model", "recommendation": "Weighted average for V1" }
  ],
  "blockers": [
    { "type": "credential", "description": "Missing Clay API key", "severity": "blocking" }
  ],
  "acceptance_criteria": ["Partners are scored 0-100", "Tiers are assigned based on score ranges"],
  "mockup_reference": null,
  "prior_artifacts": null
}
```

### Versioning

Table: `project_description_versions`

| Column | Type | Description |
|--------|------|-------------|
| project_description_id | UUID | FK to project_descriptions |
| version | INT | Version number |
| snapshot | JSONB | Full state at this version |
| change_summary | TEXT | What changed |
| change_source | TEXT | 'planner', 'user_edit', 'chat_agent', 'execution_result' |
| changed_by | TEXT | Who made the change |

### Issues / Blockers

Table: `node_issues`

| Column | Type | Description |
|--------|------|-------------|
| node_id | UUID | FK to execution_nodes |
| session_id | UUID | FK to execution_sessions |
| issue_type | TEXT | 'credential', 'manual', 'decision', 'external', 'technical' |
| description | TEXT | Human-readable description |
| status | TEXT | 'open', 'resolved', 'dismissed' |
| source | TEXT | 'preflight', 'agent', 'user', 'system' |

### Threads (Section-Level Discussion)

Table: `description_threads`

| Column | Type | Description |
|--------|------|-------------|
| session_id | UUID | FK to execution_sessions |
| node_id | UUID | Optional FK to execution_nodes |
| section_path | TEXT | e.g., "architecture", "io_contract" |
| highlighted_text | TEXT | Optional text that was highlighted |
| status | TEXT | 'open', 'resolved', 'archived' |

Table: `description_thread_messages`

| Column | Type | Description |
|--------|------|-------------|
| thread_id | UUID | FK to description_threads |
| role | TEXT | 'user' or 'assistant' |
| content | TEXT | Message content |
| description_patch | JSONB | Optional proposed changes to the node description |

## Part 5: Editing Protocol

### Manual Section Editing

1. User clicks pencil icon on a section in the document view.
2. Textarea appears with current content.
3. User edits and clicks Save.
4. Frontend calls `PATCH /api/execute/:session_id/nodes/:node_id/description` with the updated description JSONB.
5. Backend updates the node and creates a version snapshot with `change_source: 'user_edit'`.

### AI-Assisted Section Editing

1. User clicks "Edit with AI" on a section header.
2. `InlineChatInput` appears. User types an instruction (e.g., "add rate limiting to the technical spec").
3. Frontend creates a thread: `POST /api/execute/:session_id/threads` with `{ node_id, section_path }`.
4. Frontend posts the instruction: `POST .../threads/:thread_id/messages` with `{ message }`.
5. Backend calls the LLM with thread context + current description. LLM responds with content + optional `description_patch`.
6. Frontend shows `DiffPreview` with original vs. proposed changes.
7. On accept: `PATCH .../nodes/:node_id/description` with the merged description.
8. On reject: dismiss, no mutation.

### Section-Level Commenting

> **Status: Backend implemented, frontend not implemented.** The `description_threads` and `description_thread_messages` tables exist (migration 025), and API endpoints for thread CRUD are registered. However, no `CommentSidebar` component exists in the frontend — threads are not exposed in the UI.

**Planned behavior:**
1. User clicks comment icon on a section header.
2. Thread is created: `POST /api/execute/:session_id/threads` with `{ node_id, section_path }`.
3. CommentSidebar opens showing the thread.
4. User types a comment/question.
5. `POST .../threads/:thread_id/messages` sends the message and triggers an LLM reply.
6. Conversation continues in the thread. Agent may propose description changes via `description_patch`.

### Queue-Based Orchestrator Steering

1. User types a message in the orchestrator input box.
2. Message is persisted to `node_messages` on the master node.
3. If orchestrator is idle: it processes immediately.
4. If orchestrator is busy: message is queued. `user_message_queued` SSE event is emitted.
5. At the next break point, orchestrator reads queued messages and incorporates them.

## Part 6: How Rich Plan Maps to Execution

### Plan Generation

`plan_rich_description()` in `planner.rs` calls the Anthropic API with the `RICH_PLANNER_SYSTEM_PROMPT` and returns a `RichPlanOutput`:

```
RichPlanOutput {
    title: String,
    summary: String,
    architecture: Value,
    data_flows: Value,
    components: Vec<RichPlannedNode>,
}
```

Each `RichPlannedNode` has `agent_slug`, `task_description`, `depends_on`, and a flexible `description: Value` containing the structured specification.

### Conversion to Execution Nodes

`rich_plan_to_execution_nodes()` converts the rich plan to execution-ready rows:
- `agent_slug`, `task_description`, `depends_on` are extracted for the work queue.
- The full `description` JSONB is stored on the `execution_nodes` row.
- `acceptance_criteria` from the description is mapped to `execution_nodes.acceptance_criteria`.
- The project-level fields (title, summary, architecture, data_flows) are stored in `project_descriptions`.

### Proactive Blocker Identification

After plan generation, `identify_blockers()` runs:
1. For each component, examines `description.technical_spec.tools` (platform names).
2. Queries `client_credentials` and `integrations` to check if required credentials exist.
3. Creates `node_issues` with `issue_type = 'credential'`, `source = 'preflight'` for missing credentials.
4. Creates `node_issues` with `issue_type = 'manual'` for tools requiring manual setup.

These issues are visible in the plan document via `IssueCard` in `NodeSection` and issue counts in `DocumentHeader`.
